# 1.2 认识Redis

Redis诞生于2009年全称是远程词典服务器(Remote Dictionary Server)，是一个基于内存的键值型NoSQL数据库。

特点：

  1. 键值型key-value，value支持多种不同的数据结构，功能丰富
  2. 单线程，每个命令具备原子性
  3. 低延迟、速度快(基于内存、IO多路复用、良好的编码)
  4. 支持数据持久化
  5. 支持主从集群、分片集群
  6. 支持多语言客户端

# 1.3 Redis安装

  1. 安装Redis依赖

    Redis基于C语言编写，首先需要安装Redis所需要的GCC依赖

    yum install -y gcc tcl

  2. 获取redis源码

    下载地址 (opens new window)

  3. 解压

    tar -zxvf redis-x.x.x.tar.gz

  4. 进入解压目录并执行

    make && make install

  5. 进入安装目录备份配置文件

    cp redis.conf redis.conf.back

  6. 修改配置文件

    # 默认为127.0.0.1只能本地访问
bind 0.0.0.0

# 守护进程的方式启动
daemonize yes

# 密码
requirepass 123456

# 当前运行redis-server的工作目录
dir /usr/local/src/redis-6.2.7

# redis能够使用的最大内存
maxmemory 512mb

# 日志文件,默认为空,不记录日志,可以指定日志文件名
logfile "redis.log"

  7. 启动Redis

    redis-server redis.conf

  8. 注册为系统服务

    新建配置文件

    vi /etc/systemd/system/redis.service

    内容如下

    [Unit]
Description=redis-server
After=network.target

[Service]
Type=forking
ExecStart=/usr/local/bin/redis-server /usr/local/src/redis-6.2.7/redis.conf

[Install]
WantedBy=multi-user.target

    重新加载服务

    systemctl daemon-reload

    启动服务

    systemctl start redis

# 1.4.1 Redis命令行客户端

Redis安装完成后就自带了命令行客户端：redis-cli，使用方式如下：

redis-cli [options][commonds]

其中常见的optons有：

  • -h 127.0.0.1：指定连接的redis节点的IP地址，默认是127.0.0.1
  • -p 6379：指定要连接的redis节点的端口，默认6379
  • -a 123456：指定redis的访问密码

其中commonds就是Redis的操作命令，例如：

  • ping：与redis服务端做心跳测试，服务端正常会返回pong

不指定commond时，会进入redis-cli的交互控制台

Redis是一个key-value的数据库，key一般是String类型，不过value的类型多种多样

# 2.3 String类型

String类型，也就是字符串类型，是Redis中最简单的存储类型，其value是字符串，不过根据字符串的格式不同，又可以分为3类

类型 说明 key value
string 普通字符串 msg hello world
int 整数类型，可以做自增、自减操作 num 10
float 浮点类型，可以做自增、自减操作 score 92.5

不管是那种类型，底层都是字节数组形式存储，只不过是编码方式不同。字符串类型的最大空间不能超过512m

  • set：添加或者修改已经存在的一个String类型的键值对

    # 设置msg的值为hello world有空格使用双引号包裹
SET msg "hello world"

  • get：根据key获取String类型的value

    # 获取key为msg的值
GET msg

  • mset：批量添加键值对

    # 添加name的值为zhangsan，num的值为10
MSET msg "hello world" num 10

  • mget：批量获取键值对

    # 获取msg和num的值
MGET msg num

  • incr：让一个整型key自增1

    # num加1
INCR num

  • incrby：让一个整型key自增并指定步长

    # num加10
INCRBY num 10

  • decr：让一个整型key自减1

    # num减1
DECR num

  • incrbyfloat：让一个浮点型的key自增并指定步长

    # socre加10
INCRBYFLOAT socre 10

  • setnx：添加一个String类型的键值对，前提是这个key不存在，否则不执行

    # 设置msg的值为hello world
SETNX msg "hello world"

  • setex：添加一个String类型的键值对，并且指定有效期

    # 设置num的值为300过期时间为20s
SETEX num 20 300

# 2.4 Hash类型

Hash类型，也叫散列，其value是一个无序字典，类似于Java中的HashMap结构

String结构是将对象序列化为JSON字符串后存储，当需要修改对象某个字段时很不方便：

KEY VALUE
meinil:user:1 {name: "Jack", age: 21}
meinil:user:2 {name:"Rose", age:18}

Hash结构可以将对象中的每个字段独立存储，可以针对单个字段做CRUD：

KEY VALUE
field value
meinil:user:1 name Jack
age 21
meinil:user:2 name Rose
age 18
- `hset`：添加或者修改hash类型key的field值 
# 设置meinil:user:1的name字段的值为Jack
HSET meinil:user:1 name Jack

  • hget：获取hash类型某一字段的值

    # 获取meinil:user:1的name字段
HGET meinil:user:1 name

  • hmset：批量设置hash值

    HMSET meinil:user:1 name Jack age 21 score 83.2

  • hmget：批量获取hash的字段值

    HMGET meinil:user:1 name age score

  • hgetall：获取某个hash类型所有字段值

    HGETALL meinil:user:1

  • hkeys：获取一个hash类型的key中所有的field

    HKEYS meinil:user:1

  • hvals：获取一个hash类型的key中的所有value

    HVALS meinil:user:1

  • hincrby：让一个hash类型key的字段值自增并指定步长

    HINCRBY meinil:user:1 age 10

  • hsetnx：添加一个hash类型的key的field值，前提是这个field不存在，否则不执行

    HSETNX meinil:user:1 sex man

# 2.5 List类型

Redis中的List类型与Java中的LinkedList类似，可以看做是一个双向链表结构。既可以支持正向检索也可以支持反向检索

特征与LinkedList类似

  • 有序
  • 元素可以重复
  • 插入和删除速度快
  • 查询速度一般

常用来存储一个有序数据，例如：朋友圈点赞列表，评论列表等

List常用命令

  • lpush：向列表左侧插入一个或多个元素

    # 向key为user的左侧插入jack、rose
LPUSH user jack rose

  • lpop：移除并返回列表左侧的第一元素，没有则返回nil

    # 从key为user的左侧移除一个元素
LPOP user

  • rpush：向列表右侧插入一个或多个元素

    # 向key为user的右侧插入alice
RPUSH user alice

  • rpop：移除并返回列表右侧的第一个元素

    # 从key为user的右侧移除一个元素
RPOP user

  • lrange：返回一段交表范围内的所有元素，下标从0开始

    # 从key为user中取出范围1到2内的元素
LRANGE user 1 2

  • blpop和brpop：与lpop和rpop类似，只不过在没有元素时等待指定时间，而不是直接返回nil

    # 从key为user的左侧移除一个元素,如果没有则等待100s
BLPOP user 100

# 2.6 Set类型

Redis的Set结构与java中的HashSet类似，可以看做是一个value为null的HashMap。因为它也是一个hash表，因此具备与HashSet类似的特征：

  • 无序
  • 元素不可重复
  • 查找快
  • 支持交集、并集、差集等功能

Set常用命令

  • sadd：向set中添加一个或多个元素

    # 给张三和李四添加好友
SADD friends:zhangsan lisi wangwu zhaoliu
SADD friends:lisi wangwu mazi ergou

  • srem：移除set中的指定元素

    # 移除张三的好友王五
SREM friends:zhangsan wangwu

  • scard：返回set中元素的个数

    # 查询张三的好友总数
SCARD friends:zhangsan

  • sismember：判断一个元素是否存在于set中

    # 判断李四是否张三好友
SISMEMBER friends:zhangsan lisi

  • smembers：获取set中的所有元素

    # 查询张三的所有好友
SMEMBERS friends:zhangsan

  • sinter：求两个集合的交集

    # 查询张三和李四的共同好友
SINTER friends:zhangsan friends:lisi

  • sdiff：求两个集合的差集

    # 查询是张三而不是李四的好友
SDIFF friends:zhangsan friends:lisi

  • sunion：求两个集合的并集

    # 查询张三和李四的所有好友
SUNION friends:zhangsan friends:lisi

# 2.7 SortedSet类型

Redis的SortedSet是一个可排序的set集合，与Java中的TreeSet有些类似，但底层数据结构差别很大。SortedSet中的每一个元素都带有一个score属性，可以基于score属性对元素排序，底层的实现是一个跳表SkipList加hash表

SortedSet具备如下特性：

  • 可排序
  • 元素不重复
  • 查询速度快

因为SortedSet的可排序特性，经常被用来实现排行榜这样的功能

SortedSet命令

  • zadd：添加一个或多个元素到SortedSet，如果已经存在则更新其score值

    # 向student中添加元素并指定score值
ZADD student 85 Jack 89 Lucy 82 Rose 95 Tom 78 Jerry 92 Amy 76 Miles

  • zrem：删除SortedSet中的一个指定元素

    # 删除student中的Tom
ZREM student Tom

  • zscore：获取SortedSet中的指定元素的score值

    # 获取Rose的score值
ZSCORE student Rose

  • zrank：获取SortedSet中的指定元素的排名

    # 获取Rose的排名
 ZRANK student Rose

  • zcard：获取SortedSet中的元素个数

    # 获取student中的元素个数
ZCARD student

  • zcount：统计score值在给定范围内的所有元素的个数

    # 统计student中score在0~80的个数
ZCOUNT student 0 80

  • zincrby：让SortedSetz中的指定元素自增，步长为指定的increment值

    # 给Rose同学的socre加2分
ZINCRBY student 2 Rose

  • zrange：按照score排序后，获取指定排名范围内的元素

    # 查找前三名
ZREVRANGE student 0 2

  • zrangebyscore：按照score排序后，获取指定socre范围内的元素

    # 查询80分以下的同学
ZRANGEBYSCORE student 0 80

  • zdiff、zinter、zunion：求差集、交集、并集

🌟所有的排名默认都是升序，如果要降序则在命令的Z后面添加REV即可

# 3.1.2 Jedis连接池

Jedis本身是线程不安全的，并且频繁的创建和销毁连接会有性能损耗，因此一般使用Jedis连接池来代替Jedis的直连方式

  1. 创建连接池工具类

    public class JedisConnectionFactory {
    private final static JedisPool JEDIS_POOL;

    static {
        JedisPoolConfig config = new JedisPoolConfig();
        // 最大连接
        config.setMaxTotal(8);
        // 最大空闲连接
        config.setMaxIdle(8);
        // 最小空闲连接
        config.setMinIdle(0);
        // 设置最长的等待时间 ms
        config.setMaxWaitMillis(200);

        JEDIS_POOL = new JedisPool(config, "192.168.3.157", 6379, 1000, "123456");
    }

    public static Jedis getJedis() {
        return JEDIS_POOL.getResource();
    }
}

  2. 使用

    public class JedisTest {
    private Jedis jedis;

    @BeforeEach
    public void before() {
        // 1. 建立连接
        jedis = JedisConnectionFactory.getJedis();

        // 2. 选择库
        jedis.select(0);
    }

    @AfterEach
    public void after() {
        // 关闭连接
        if (jedis != null) {
            jedis.close();
        }
    }

    @Test
    public void testString() {
        jedis.set("name", "张三");
        System.out.println(jedis.get("name"));
    }

}

# 3.2.3 序列化

RedisTemplate可以接收任意Object作为值写入Redis，只不过写入前会把Object序列化为字节形式，默认是JDK的序列化方式，以上述为例redis中存储的结果为

# key
\xac\xed\x00\x05t\x00\x0dredisTemplate

# value
\xac\xed\x00\x05t\x00\x0aSpringData

配置RedisTemplate的序列化方式：key采用String，值序列化为String或者JSON

@Configuration
public class RedisConfig {
    @Bean
    public RedisTemplate<String, Object> redisTemplate(RedisConnectionFactory connectionFactory) {
        // 创建RedisTemplate
        RedisTemplate<String, Object> redisTemplate = new RedisTemplate<>();
        // 设置连接工厂
        redisTemplate.setConnectionFactory(connectionFactory);

        // 设置序列化工具
        GenericJackson2JsonRedisSerializer serializer = new GenericJackson2JsonRedisSerializer();

        // key和hashKey采用String序列化
        redisTemplate.setKeySerializer(RedisSerializer.string());
        redisTemplate.setHashKeySerializer(RedisSerializer.string());

        // value和hashValue采用JSON序列化
        redisTemplate.setValueSerializer(serializer);
        redisTemplate.setHashValueSerializer(serializer);

        return redisTemplate;
    }
}

测试

class RedisTest {

为了在反序列化时知道对象的类型，JSON序列化器会将类的全类名写入json结果并存入Redis中，会带来额外的内存开销

# key
user:100

# value
{
    "@class": "com.meinil.redis.User",
    "name": "二狗",
    "age": 30
}

为了节省内存空间，我们并不会使用JSON序列化器来处理value，而是统一使用String序列化器，要求只能存储String类型的key和value。当需要存储Java对象时，手动完成对象的序列化和反序列化

Spring默认提供了一个StringRedisTemplate类，它的key和value的序列化方式默认就是String方式。

@SpringBootTest
class RedisTest {
    @Autowired
    private StringRedisTemplate stringRedisTemplate;

    @Test
    public void testStringRedisTemplate() throws JsonProcessingException {
        ObjectMapper mapper = new ObjectMapper();
        User user = new User("二狗", 30);
        
        // 将对象序列化为JSON字符串后存入
        stringRedisTemplate.opsForValue().set("user:200", mapper.writeValueAsString(user));

        String value = stringRedisTemplate.opsForValue().get("user:200");
        
        // 反序列化
        User user1 = mapper.readValue(value, User.class);
        System.out.println(user1);
    }
}
@Data
@NoArgsConstructor
@AllArgsConstructor
class User {
    private String name;
    private Integer age;
}

# 4.2.2 缓存更新策略

内存淘汰 超时剔除 主动更新
说明 不用自己维护，利用Redis的内存淘汰机制，当内存不足时自动淘汰部分数据。下次查询时更新缓存 给缓存数据添加TTL时间，到期后自动删除缓存。下次查询时更新缓存 编写业务逻辑，在修改数据库的同时，更新缓存
一致性 差 一般 好
维护成本 无 低 高
  • 低一致性需求：使用内存淘汰机制。例如店铺类型的查询缓存
  • 高一致性需求：主动更新，并以超时剔除作为兜底方案。例如店铺详情查询的缓存

主动更新策略

  1. 由缓存的调用者，在更新数据库的同时更新缓存

    • 删除缓存还是更新缓存

      更新缓存：每次更新数据库都更新缓存，无效写操作较多

      删除缓存🌟：更新数据库时让缓存失效，查询时再更新缓存

    • 如何保证缓存与数据库的操作的同时成功或失败

      单体系统，将缓存与数据库操作放在一个事务

      分布式系统，利用TCC等分布式事务方案

    • 先操作缓存还是先操作数据库

      先删除缓存，再操作数据库

      🌟先操作数据库，再删除缓存

  2. 缓存与数据库整合为一个服务，由服务来维护一致性。调用者调用改服务，无序关心缓存一致性问题

  3. 调用者只操作缓存，由其他线程异步的将缓存持久化到数据库，保证最终一致

  • 时间戳：31bit，可以使用69年

@Component
public class RedisIdWorker {

    /**
     * 开始时间戳
     */
    private final static long BEGIN_TIMESTAMP = 1640995200L;

    /**
     * 序列号位数
     */
    private final static int COUNT_BITS = 32;

    private final static DateTimeFormatter DATE_TIME_FORMATTER = DateTimeFormatter.ofPattern("yyyy:MM:dd");

    @Autowired
    private StringRedisTemplate redisTemplate;

    public long nextId(String keyPrefix) {
        // 1. 生成时间戳
        LocalDateTime now = LocalDateTime.now();

        long nowSecond = now.toEpochSecond(ZoneOffset.UTC);
        long timestamp = nowSecond - BEGIN_TIMESTAMP;

        // 2. 生成序列号
        long count = redisTemplate.opsForValue().increment("icr:" + keyPrefix + ":" + now.format(DATE_TIME_FORMATTER));

        // 3. 拼接并返回
        return timestamp << COUNT_BITS | count;
    }
}

  • 库存是否充足，不足则无法下单

        .eq(SeckillVoucher::getVoucherId, voucherId)

    LocalDateTime now = LocalDateTime.now();
    if (voucher.getBeginTime().isAfter(now)) {

        throw new RuntimeException("抢购时间已经结束");

        throw new RuntimeException("优惠券已经被抢光了");
    }

    voucher.setStock(voucher.getStock() - 1);

        .set(SeckillVoucher::getStock, voucher.getStock())
        .eq(SeckillVoucher::getVoucherId, voucherId)
        .update();
    if (!success) {
        throw new RuntimeException("服务器异常");
    }

    // 6. 生成订单
    VoucherOrder voucherOrder = new VoucherOrder();
    voucherOrder.setId(idWorker.nextId("order"));
    voucherOrder.setVoucherId(voucherId);
    voucherOrder.setUserId(UserHolder.getUser().getId());
    baseMapper.insert(voucherOrder);

    return voucherOrder.getId();
}

    public class SimpleRedisLock implements ILock{

    private StringRedisTemplate redisTemplate;
    private final static String KEY_PREFIX = "lock:";

        this.name = name;

        // 获取线程id
        long threadId = Thread.currentThread().getId();

        // 获取锁
        Boolean success = redisTemplate.opsForValue().setIfAbsent(KEY_PREFIX + name, String.valueOf(threadId), timeout, TimeUnit.SECONDS);
        return BooleanUtil.isTrue(success);
    }

    @Override
    public void unLock() {
        // 释放锁
        redisTemplate.delete(KEY_PREFIX + name);
    }

# 4.3.9 Redis锁误删问题

假设现在某一个线程（线程1）获取到了锁，但是由于它的业务逻辑过于复杂，就有可能在业务逻辑还未完成时，锁就过期了(提前释放了)，这时候就其他线程（线程2）就可以去抢占这个锁，当线程1业务逻辑后执行释放锁的逻辑，会导致线程2的锁被释放，造成安全问题

解决方案：在获取锁时，存入当前锁的一个标识(UUID实现)，当触发释放锁的操作时，先判断当前锁是否是自己的锁，如果是则释放，否则不做任何处理

public class SimpleRedisLock implements ILock{

    private String name;
    private StringRedisTemplate redisTemplate;
    private final static String KEY_PREFIX = "lock:";
    private final static String ID_PREFIX = UUID.randomUUID().toString(true) + "-";

    public SimpleRedisLock(String name, StringRedisTemplate redisTemplate) {
        this.name = name;
        this.redisTemplate = redisTemplate;
    }

    @Override
    public boolean tryLock(long timeout) {
        // 获取线程标识
        String threadId = ID_PREFIX + Thread.currentThread().getId();

        // 获取锁
        Boolean success = redisTemplate.opsForValue().setIfAbsent(KEY_PREFIX + name, threadId, timeout, TimeUnit.SECONDS);
        return BooleanUtil.isTrue(success);
    }

    @Override
    public void unLock() {
        // 获取线程标识
        String threadId = ID_PREFIX + Thread.currentThread().getId();

        String lock = redisTemplate.opsForValue().get(KEY_PREFIX + name);

        if (threadId.equals(lock)) {
            // 释放锁
            redisTemplate.delete(KEY_PREFIX + name);
        }
    }
}

# 4.3.10 Redis释放锁原子性问题

在上述例子中，一个线程(线程A)是先判断锁是否是自己的，然后再去释放锁，这时候会产生一个情况，如果当判断是否是自己为true后，锁立刻过期了，这时候别的线程(线程B)就可以获取到锁，线程A再去执行释放锁的操作，此时线程A释放的锁并不是自己的锁，造成了锁的误删

end

# 4.3.11 分布式锁优化问题

基于setnx实现的分布式锁存在下面的问题：

  1. 不可重入

    用一个线程无法多次获取同一把锁

  2. 不可重试

    获取锁只尝试一次就返回fasle，没有重试机制

  3. 超时释放

    锁超时释放虽然可以避免死锁，但如果是业务执行耗时较长，也会导致锁释放，存在安全隐患

  4. 主从一致性

    如果Redis提供了主从集群，主从同步存在延迟，当主宕机时，如果从没有同步已经宕机的锁的数据，则会出现重复获取锁的安全问题

# 5.3 可重入锁原理

Redisson在第一次获取锁时会进行添加操作，如果是第二次获取锁，则是在原来的基础上自增1，同时重置锁的有效期，释放锁时并非真正释放锁，而是将当前值减一之后判断是否等于0，等于0则释放锁

  1. 获取锁Lua脚本

    -- 锁的key
local key = KEYS[1]
-- 线程的唯一标识
local threadId = ARGV[1]
-- 锁的自动释放时间
local releaseTime = ARGV[2]
-- 判断锁是否存在
if (redis.call('exists', key) == 0) then
    -- 不存在,获取锁
    redis.call('hset', key, threadId, '1')
    -- 设置有效期
    redis.call('expire', key, releaseTime)
    -- 返回结果
    return 1
end

if (redis.call('hexists', key, threadId) == 1) then
    -- 不存在,获取锁,重入次数+1
    redis.call('hincrby', key, threadId, '1')
    -- 设置有效期
    redis.call('expire', key, releaseTime)
    -- 返回结果
    return 1
end

-- 代码走到这里,说明获取锁的不是自己,获取锁失败
return 0

  2. 释放锁脚本

    -- 锁的key
local key = KEYS[1]
-- 线程的唯一标识
local threadId = ARGV[1]
-- 锁的自动释放时间
local releaseTime = ARGV[2]
-- 判断当前锁是否还被自己持有
if (redis.call('hexists', key, threadId) == 0) then
    -- 当前锁不是自己的锁,直接返回
    return nil
end

-- 是自己的锁,则重入次数-1
local count = redis.call('hincrby', key, threadId, -1)
if (count > 0) then
    -- 大于0说明不能释放锁,重置有效期返回
    redis.call('expire', key, releaseTime)
    return nil
else
    -- 等于0说明可以释放锁,直接删除
    redis.call('del', key)
    return nil
end

# 5.4 联锁原理

如果在Redis集群的模式下，存在主从同步，当一个线程获取锁后，主Redis宕机，而锁还未同步至从机，则会导致锁丢失，引起并发问题

Redisson的连锁将主从关系去除，只有当在所有的节点上获取到锁后才算成功，如果有一台失败，则获取锁失败

# 6.2 消息队列

最简单的消息队列模型包括三个角色

  • 消息队列：存储和管理消息，也被称为消息代理(Message Broker)
  • 生产者：发送消息到消息队列
  • 消费者：从消息队列获取消息并处理消息

Redis提供了三种不同的方式来实现消息队列

  • list结构：基于List结构模拟消息队列
  • PubSub：基本的点对点消息模型
  • Stream：比较完善的消息队列模型

# 6.2.1 基于LIst结构

Redis的List数据结构是一个双向链表，很容易模拟出队列效果

队列是入口和出口不在一起，因此我们可以利用：LPUSH结合RPOP、或者RPUSH结合LPOP来实现

不过要注意的是，当队列中没有消息时RPOP或LPOP操作会返回null，并不会像JVM的阻塞队列那样会阻塞并等待消息，因此这里应该使用BRPOP或者BLPOP来实现阻塞效果

基于List的消息队列有哪些优缺点

优点：

  • 利用Redis存储，不受限于JVM内存上限
  • 基于Redis的持久化机制，数据安全性有保证
  • 可以满足消息有序性

缺点：

  • 无法避免消息丢失
  • 只支持单消费者

# 6.2.2 基于PubSub

PubSub(发布订阅)是Redis2.0版本引入的消息传递模型。消费者可以订阅一个或者多个Channel，生产者向对应channel发送消息后，所有订阅者都能收到相关消息

  • 订阅一个或多个频道

    SUBSCRIBE channel [channel]

  • 向一个频道发送消息

    PUBLISH channel msg

  • 订阅与pattern格式匹配的所有频道

    PSUBSCRIBE pattern [pattern]

基于PubSub的消息队列有哪些优缺点

优点：

  • 采用发布订阅模型，支持多生产、多消费

缺点：

  • 不支持数据持久化
  • 无法避免消息丢失
  • 消息堆积有上限，超出时数据丢失

# 6.2.3 基于Stream

Stream时Redis5.0引入的一种新的数据类型。可以实现一个功能非常完善的消息队列

例如：创建名为users的队列，并向其中发送一个消息，内容：{name=jack,age=21}，并且使用Redis自动生成ID

XADD users * name jack age 21

读取：$代表读取最新的消息，

XREAD COUNT 1 BLOCK 1000 STREAMS users $

注意：当我们指定起始ID为$时，代表读取最新的消息，如果我们处理一条消息的过程中，又有超过1条以上的消息到达队列，则下次获取时也只能获取到最新的一条，会出现漏对消息的问题

STREAM类型消息队列的XREAD命令特点：

  • 消息可回溯
  • 一个消息可以被多个消费者读取
  • 可以阻塞读取
  • 有消息漏读的风险

# 6.2.4 Stream消费者组

消费者组(Consumer Group)：将多个消费者划分到一个组中，监听同一个队列。具备下列特点：

  1. 消息分流

    队列中的消息会分流给组内的不同消费者，而不是重复消费，从而加快消息处理的速度

  2. 消息标示

    消费者组会维护一个标示，记录最后一个被处理的消息，哪怕消费者宕机重启，还会从标示之后读取消息。确保每一个消息都会被消费

  3. 消息确认

    消费者获取消息后，消息处于pending状态，并存入一个pending-list。当处理完成后需要通过XACK来确认消息，标记消息为已处理，才会从pending-list移除

创建消费者组：

XGROUP CREATE key groupName ID [MKSTREAM]
  • key：队列名称
  • groupName：消费者组名称
  • ID：起始ID标示，$代表队列中最后一个消息，0则代表队列中第一个消息
  • MKSTREAM：队列不存在时自动创建队列

其他常见命令：

  1. 删除指定的消费者组

    XGROUP DESTORY key groupName

  2. 给指定的消费者组添加消费者

    XGROUP CREATECONSUMER key groupName consumerName

  3. 删除消费者组中的指定消费者

    XGROUP DELCONSUMER key groupName consumerName

从消费者组读取消息：

XREADGROUP GROUP group consumer [COUNT count] [BLOCK milliseconds] [NOACK] STREAMS key [key ...] ID [ID ...]

  • group：消费组名称

  • consumer：消费者名称，如果消费者不存在，会自动创建一个消费者

  • count：本次查询的最大数量

  • BLOCK milliseconds：当没有消息时最长等待时间

  • NOACK：无需手动ACK，获取到消息后自动确认

  • STREAMS key：指定队列名称

  • ID：获取消息的起始ID

    ">"：从下一个未消费的消息开始

    其他：根据指定id从pending-list中获取已消费但未确认的消息，例如0，是从pending-list中的第一个消息开始

STREAM类型消息队列的XREADGROUP命令特点：

  • 消息可回溯
  • 可以多消费者争抢消息，加快消费速度
  • 可以阻塞读取
  • 没有消息漏读的风险
  • 有消息确认机制，保证消息至少被消费一次

        SECKILL_SCRIPT.setLocation(new ClassPathResource("lua/seckill.lua"));
        SECKILL_SCRIPT.setResultType(Long.class);

     * 阻塞队列
     */

     * 线程池
     */
    private final static ExecutorService executors = Executors.newSingleThreadExecutor();

    @PostConstruct
    private void init() {
        executors.submit(() -> {
            while (true) {
                try {
                    // 获取消息队列中的消息
                    List<MapRecord<String, Object, Object>> recordList = redisTemplate.opsForStream().read(
                            Consumer.from("g1", "c1"),
                            StreamReadOptions.empty().count(1).block(Duration.ofSeconds(2)),
                            StreamOffset.create("stream.orders", ReadOffset.lastConsumed())
                    );
                    if (CollectionUtil.isEmpty(recordList)) {
                        // 获取失败，继续下一次循环
                        continue;
                    }

                    // 解析消息
                    MapRecord<String, Object, Object> record = recordList.get(0);
                    VoucherOrder voucherOrder = BeanUtil.fillBeanWithMap(record.getValue(), new VoucherOrder(), true);

                    // 下单
                    createOrder(voucherOrder);

                    // 确认消息
                    redisTemplate.opsForStream().acknowledge("stream.orders", "g1", record.getId());
                } catch (Exception e){
                    log.error("处理异常消息");
                    handlePendingList();
                }
            }
        });
    }

    private void handlePendingList() {
        while (true) {
            try {
                // 获取pending-list中的消息
                List<MapRecord<String, Object, Object>> recordList = redisTemplate.opsForStream().read(
                        Consumer.from("g1", "c1"),
                        StreamReadOptions.empty().count(1),
                        StreamOffset.create("stream.orders", ReadOffset.from("0"))
                );
                if (CollectionUtil.isEmpty(recordList)) {
                    // pending-list 中没有异常消息 跳出循环
                    break;
                }

                // 解析消息
                MapRecord<String, Object, Object> record = recordList.get(0);
                VoucherOrder voucherOrder = BeanUtil.fillBeanWithMap(record.getValue(), new VoucherOrder(), true);

                // 下单
                createOrder(voucherOrder);

                // 确认消息
                redisTemplate.opsForStream().acknowledge("stream.orders", "g1", record.getId());
            } catch (Exception e){
                log.error("处理pending-list订单异常");
            }
        }
    }

    @Override
    public Long seckillVoucher(Long voucherId) {
        // 1. 执行lua脚本
        Long userId = UserHolder.getUser().getId();
        long orderId = idWorker.nextId("order");
        Long execute = redisTemplate.execute(
                SECKILL_SCRIPT,
                Collections.emptyList(),
                voucherId.toString(), userId.toString(), String.valueOf(orderId)
        );

        int r = execute.intValue();
        if (r == 3) {
            throw new RuntimeException("库存不足");
        }

        if (r == 2) {
            throw new RuntimeException("你已拥有此券,请不要重复下单");
        }

        if (r == 0) {
            throw new RuntimeException("未知错误");
        }

        return orderId;
    }

    @Override
    @Transactional(rollbackFor = Exception.class)
    public void createOrder(VoucherOrder voucherOrder) {
        Long userId = voucherOrder.getUserId();

        // 1. 判断当前用户是否已经有券
        Integer count = baseMapper.selectCount(
                Wrappers.lambdaQuery(VoucherOrder.class)
                        .eq(VoucherOrder::getVoucherId, voucherOrder.getId())
                        .eq(VoucherOrder::getUserId, userId)
        );
        if (count > 0) {
            throw new RuntimeException("你已抢过此券");
        }

        // 2. 扣减库存
        boolean success = seckillVoucherService.lambdaUpdate()
                .setSql("stock = stock - 1")
                .eq(SeckillVoucher::getVoucherId, voucherOrder.getVoucherId())
                .gt(SeckillVoucher::getStock, 0)
                .update();
        if (!success) {
            throw new RuntimeException("服务器异常");
        }

        // 3. 插入订单
        baseMapper.insert(voucherOrder);
    }
